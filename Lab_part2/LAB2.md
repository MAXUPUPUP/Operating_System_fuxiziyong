# 操作系统配套实验 · 模块二（进程·线程·调度）

对应第 4~5 课。三个实验:线程的共享/私有、竞态条件、调度的不确定性。
全部基于 POSIX `pthread`,**Linux 与 macOS 通用**。

## 环境与编译
- macOS:先 `xcode-select --install` 装好命令行工具
- pthread 程序**必须加 `-pthread`**（Makefile 里已加好）

```bash
make            # 编译出 labA labB labC
./labA          # 运行
make clean
```
> 若 macOS 上 `gcc` 实为别名报错,把 Makefile 里 `CC = gcc` 改成 `CC = cc` 即可。

## Lab A — 共享全局 vs 私有栈
**文件:** `labA_pthread.c`　**对应:** 第四课

```bash
./labA
```
**观察:**
- 所有线程打印的 `g_shared` 地址**完全相同** → 全局变量是**共享**的。
- 各线程 `local` 变量的地址**各不相同** → 每个线程有**自己独立的栈**。
- 这就是第四课那张图的实证:线程共享进程级资源,独占自己的栈。

**改改看:**
- 让某个线程修改 `g_shared`(如 `g_shared += id;`),其它线程会看到变化吗?(会 —— 因为共享。但注意:多个线程同时改就有 Lab B 的问题了。)

## Lab B — 竞态条件,以及锁怎么修好它（核心）
**文件:** `labB_race.c`　**对应:** 第四课结尾 + 第五课

```bash
./labB          # 多跑几次
```
**观察:**
- 4 个线程各把 `counter` 自增 10 万次,**期望 = 400000**。
- **不加锁**:结果远小于 40 万,而且**每次都不一样** —— 因为 `counter++` 的"读→改→写"三步被多个线程交错,旧值覆盖新值,更新丢失。这叫 **race condition(竞态条件)**。
- **加锁**:用互斥锁把这三步保护成不可分割的整体,结果**每次都精确 400000**。

**代码里的关键:** 我故意把 `counter++` 显式拆成
`tmp = counter; (停顿一下); counter = tmp + 1;`
并在中间留了一道缝,好让单核机器也能稳定复现 race。真实的 `counter++` 在多核上同样会出错,只是窗口更窄、更隐蔽。

**这条最重要的教训:** race 是**不确定**的 bug —— 它可能跑一万次都对,偏偏某次就错。所以并发正确性**不能靠"测着没问题"来保证**,必须靠正确的同步(加锁)从机制上杜绝。这正是下一个模块(并发与同步)要系统讲的。

**改改看:**
1. 把中间那行 `for (volatile int k...)` 删掉,看不加锁还出不出错(窗口变窄,出错变少但不代表对)。
2. 把锁换成只锁"写回"一步,看还对不对(体会:必须把整个读-改-写都锁住才行)。

## Lab C — 调度的不确定性
**文件:** `labC_schedule.c`　**对应:** 第五课

```bash
./labC          # 连跑几次
```
**观察:**
- 3 个线程各打印 5 步,输出**交错**在一起 —— 谁先谁后由**调度器**决定。
- **多跑几次,顺序会变**:这说明线程的执行顺序是操作系统调度的结果,程序自己控制不了。
- 这也解释了 Lab B 为什么"每次结果不同":同样是调度时机的不确定性在起作用。

**改改看:**
- 把 `usleep(1000)` 删掉或调大,观察交替的颗粒度变化。
- 线程数从 3 改成 8,看交错是不是更密。

## 这套实验把模块二串起来了
- **第四课**:线程共享全局/堆、独占栈(Lab A);共享带来的代价 —— 竞态(Lab B)。
- **第五课**:调度决定执行顺序且不确定(Lab C),这正是竞态难复现、难调试的根源。
- **承上启下**:Lab B 留下的"病"(race),下个模块用"药"(锁、信号量)系统地治。

> 单核小贴士:本套在单核环境也能复现 race(靠 Lab B 里那道人为的缝)。
> 你在多核 Mac 上跑,race 会更"自然"地出现,可以把那道缝删了再试。
